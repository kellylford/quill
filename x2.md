# QUILL Startup Wizard — Design Plan

**Purpose:** First-run (and re-run) wizard that lets users shape QUILL into their editor.
Features they disable are completely hidden — no menus, no commands, no noise.
**Audience:** This document is the authoritative implementation spec.

---

## 1. Goals

1. Every user gets a QUILL that fits their work and accessibility needs from minute one.
2. Disabled features disappear from menus, the command palette, and keyboard shortcuts.
3. The wizard is accessible by keyboard and screen reader from the first keystroke.
4. Re-running the wizard at any time is a single command in the Help menu.
5. No feature is permanently locked out; any choice can be reversed.

---

## 2. High-Level Architecture

```
quill/
  ui/
    setup_wizard.py          -- wx.adv.Wizard subclass; all pages
    setup_wizard_pages.py    -- individual WizardPage classes
  core/
    feature_flags.py         -- FeatureFlags dataclass, load/save helpers
    feature_registry.py      -- command-to-feature-flag mapping table
  tools/
    check_feature_tags.py    -- CI gate: every command must declare its feature tag
```

The wizard writes a `feature_flags` block into `settings.json` via the existing
`write_json_atomic` path. On startup `MainFrame.__init__` loads flags before
building menus, the command registry, and keybindings.

---

## 3. Feature Flag Schema

New block added to `settings.json` (validated by existing schema infrastructure):

```json
{
  "setup_wizard_completed": true,
  "feature_flags": {
    "ai":           true,
    "remote":       false,
    "quillins":     true,
    "power_tools":  true,
    "notebook":     false,
    "collaboration": true
  }
}
```

`FeatureFlags` is a frozen dataclass in `quill/core/feature_flags.py`.
`is_enabled(flag: str) -> bool` is the single access point everywhere.

When `setup_wizard_completed` is absent or `False`, the wizard runs at startup
before `MainFrame` is shown.

---

## 4. Wizard Pages — Full UX Specification

The wizard is a `wx.adv.Wizard`. Each page is a `wx.adv.WizardPageSimple`.
Navigation: **Back**, **Next / Finish**, **Cancel** (with confirmation).
All heading text is read as a live region when the page loads so screen readers
announce the page title without user action.

---

### Page 1 — Welcome

**Heading:** "Welcome to QUILL"

**Body text (read as static text, no heading markup so it flows naturally under
the page title):**

> QUILL is a screen-reader-first writing editor. This two-minute setup wizard
> personalises your copy so that only the features you want appear in the menus
> and command palette.
>
> You can change any of these choices later from the Tools menu — just run
> "Personalise QUILL" to come back here.
>
> Press Next to begin.

**Controls:** Next button only (no Back on page 1). Cancel asks "Exit without
setting up? QUILL will run with default settings." with Yes / No.

**Accessibility notes:**
- Focus lands on the Next button when the page loads.
- The static text block has `SetName("Welcome message")` so a screen reader
  user who tabs backward to it hears a meaningful label.

---

### Page 2 — Keyboard and Sound

**Heading:** "Keyboard and Sound"

**Design principle:** QUILL is screen-reader-first. It detects NVDA, JAWS, and
other assistive technology automatically at runtime via `sr_detect.py` and
behaves accessibly by default — no declaration required. This page asks only
about genuine personal preferences that any user might want to adjust.
Asking "are you using a screen reader?" would imply QUILL needs to be told
to be accessible; it does not.

**Control 1 — "QUILL key"**
A `wx.Choice` labelled "QUILL key (used for editor commands)":
- Caps Lock
- Insert
- None — I'll use standard shortcuts only

QUILL silently pre-selects a sensible default based on what `sr_detect.py`
detects (Caps Lock if NVDA is active, Insert if JAWS is active, Caps Lock
otherwise). The choice is shown to the user because it is a real preference —
sighted power users may want Caps Lock too, and JAWS users sometimes prefer
Insert for numpad compatibility.

A static text line below the choice updates as the selection changes:
"[Key name]: [one sentence on what this key unlocks, e.g. 'Caps Lock becomes
your QUILL key — it no longer toggles caps.']"

If a screen reader is detected, a quiet informational line (not a question)
appears above the choice: "QUILL detected [screen reader name] and has
pre-selected a compatible default." This is informational only; the user is
free to change it.

**Control 2 — "Sound effects"**
A `wx.CheckBox` labelled "Play sound effects for actions (open, save, alerts)".
Checked by default for everyone.

**Accessibility notes for this page:**
- The QUILL key choice fires `EVT_CHOICE` and immediately updates the
  explanation so the user hears what that key does before pressing Next.
- The detected screen reader line is a `wx.StaticText`; it is part of the
  natural tab/read order and requires no special announcement.
- No control on this page is gated behind a disability disclosure.

---

### Page 3 — AI Writing Assistance

**Heading:** "AI Writing Assistance"

**Gate decision — radio group "Would you like AI writing assistance?":**
- Yes, connect to an AI provider
- No, I'll write without AI

If **No** is selected:
- The rest of this page collapses (all child controls hidden, sizer re-laid out).
- `feature_flags.ai` is set to `False`.
- The page announces: "AI features will be hidden. You can enable them later
  from Settings."

If **Yes** is selected, three sub-sections become visible:

**Sub-section A — Provider**
`wx.Choice` labelled "AI provider":
- OpenAI  (cloud, requires API key)
- OpenRouter  (cloud, many models, requires API key)
- Ollama  (local, free, requires Ollama running on your machine)
- Set up later

When the choice changes, sub-sections B and C update immediately (same
per-provider model memory pattern already used in the Ask Quill dialog).
The announcement reads: "Selected [provider]. [Brief one-line description]."

**Sub-section B — API Key** (shown only for OpenAI / OpenRouter)
`wx.TextCtrl(style=wx.TE_PASSWORD)` labelled "API key".
Below it: a `wx.Button` labelled "Reveal key" toggles `wx.TE_PASSWORD`.
Below the button: a static text hint "Your key is stored in Windows Credential
Manager, never in any file."

**Sub-section C — Local host** (shown only for Ollama)
`wx.TextCtrl` labelled "Ollama host" with default value `http://localhost:11434`.
Static text hint: "Make sure Ollama is running before pressing Next."

**Sub-section D — Default model**
`wx.TextCtrl` labelled "Default model". Pre-filled with the provider's
recommended default (same `default_model_for_provider()` logic used elsewhere).
The field is always shown when a real provider is selected; hidden for
"Set up later".

**Accessibility notes for this page:**
- Hiding/showing sub-sections uses `sizer.Show(ctrl, show)` + `Layout()` so the
  tab order is always correct and no hidden control receives focus.
- The live region for the provider choice change announcement is the static
  description text immediately below the choice, which is re-populated and
  then `SetFocus()`-ed for one keystroke so NVDA/JAWS reads it, then focus
  returns to the choice.

### Page 4 — Remote File Editing

**Heading:** "Remote File Editing"

**Gate decision — radio group "Do you edit files on remote servers?":**
- Yes, I use SSH or SFTP
- No, I work with local files only

If **No**: `feature_flags.remote = False`. Page collapses to a single
confirmation line: "Remote editing will be hidden from menus."

If **Yes**:
- Static text: "QUILL can open, edit, and save files directly over SSH/SFTP.
  You can add saved sites now or later from the Remote menu."
- `wx.CheckBox` labelled "Add a saved site now". Default: unchecked.
- When checked, a mini site-entry form (Friendly name, Host, Port, Username)
  appears inline. This is not mandatory — the user can add sites later from
  the Remote menu or via Manage Remote Sites.
- The mini form uses the same factory-lambda `row()` pattern as the full
  `SiteEditDialog` so z-order is correct from the start.

**Accessibility notes:**
- The inline site form appears/disappears based on the checkbox via the same
  show/hide sizer pattern as Page 3.
- Tabbing into the collapsed area is impossible (no hidden focusable controls).

### Page 5 — QUILL Extensions (Quillins)

**Heading:** "Extensions"

**Explanation (static text):**
> QUILL extensions (Quillins) add capabilities like word count indicators,
> custom shortcuts, AI writing prompts, and more. Bundled extensions include
> an AI Prompt Pack and a Word Count sidebar.

**Gate decision — radio group "Would you like to enable extensions?":**
- Yes, enable extensions
- No, keep it simple

If **No**: `feature_flags.quillins = False`. Extensions menu and all
extension-contributed commands are hidden.

If **Yes**:
- `wx.CheckBox` labelled "Auto-install bundled extensions" (checked by default).
  If checked, the bundled Quillins under `quill/quillins_bundled/` are enabled.
- Static list (read-only `wx.ListCtrl` with two columns: Name, Description)
  shows the bundled extensions so the user knows what they are getting.
  Screen reader users can navigate the list with arrow keys; each row reads
  "Name. Description."

### Page 6 — Power Tools

**Heading:** "Power Tools"

**Explanation (static text):**
> Power Tools are advanced text operations: sort lines, shuffle text, strip
> HTML, encode/decode entities, search across files, and more. They live in
> the Power Tools menu and the command palette.

**Gate decision — radio group "Include Power Tools?":**
- Yes, I want these tools
- No, keep the menus focused

If **No**: `feature_flags.power_tools = False`. Power Tools menu and all

This page is short by design. After the radio selection there is nothing else
to configure; the user presses Next immediately.

### Page 7 — Notebook Workspace

**Heading:** "Notebook Workspace"

**Explanation (static text):**
> The Notebook workspace lets you work on a collection of related documents
> together — a research notebook, a book project, or a set of notes. Each
> document in the notebook can link to others.

**Gate decision — radio group "Enable the Notebook workspace?":**
- Yes, I work on multi-document projects
- No, I work one document at a time

If **No**: `feature_flags.notebook = False`. The Notebook menu entry and
all Notebook commands are hidden.

---

### Page 8 — Keyboard Profile

**Heading:** "Keyboard Shortcuts"

**Explanation (static text):**
> Choose a starting keyboard profile. You can fine-tune individual bindings
> at any time from Settings > Keyboard.

**Radio group "Keyboard profile":**
- QUILL Default — all built-in bindings active
- Minimal — only File, Edit, and Format shortcuts; all advanced chords off
- Screen Reader Friendly — remaps conflicts with common NVDA/JAWS shortcuts

Each radio button, when focused, triggers a brief description line below:
"[Profile name]: [one sentence description of what changes]."

The profile is stored as `keymap_profile` in settings and applied by the
keymap loader at startup. Profiles are JSON files in `quill/core/keymap/`.
The wizard page lists them by scanning that directory, so future profiles
appear automatically.

**Accessibility notes:**
- The description update on focus (not just on selection) means arrow-key
  browsing of the radio group gives immediate auditory feedback about each
  option before committing.

### Page 9 — Summary and Finish

**Heading:** "Your QUILL Setup"

**Purpose:** Give the user a clear, auditable summary of what they chose
before anything is written to disk.

**Body:** A `wx.ListCtrl` (report view, two columns: Feature, Status) listing
every decision made in pages 2-8:

| Feature              | Status         |
|----------------------|----------------|
| Screen reader        | NVDA           |
| QUILL key            | Caps Lock      |
| Sound effects        | On             |
| AI assistance        | OpenAI         |
| Remote editing       | Disabled       |
| Extensions           | Enabled        |
| Power Tools          | Enabled        |
| Notebook workspace   | Disabled       |
| Keyboard profile     | QUILL Default  |

The list is read-only and navigable by arrow keys. Each row announces
"Feature: [name]. Status: [value]." when focused.

Below the list, a single static text line:
"Press Finish to apply these settings and open QUILL."

**Back button** returns to Page 8 so the user can change anything.
**Finish button** (replaces Next on the last page) writes settings and
closes the wizard. `setup_wizard_completed = True` is written atomically.

After the wizard closes, `MainFrame._apply_feature_flags()` is called to
rebuild menus, refresh the command palette, and re-bind the keymap. If this
is first-run, QUILL then shows an empty document with the cursor ready.

## 5. Feature Flag Enforcement

### 5.1 Menu Filtering

`main_frame_menu.py` calls `is_enabled(flag)` before appending each feature-
group's menu items. Items are not created at all (not just hidden) when their
flag is off. This keeps the menu object model clean and ensures screen readers
see no phantom items.

Each menu item has a `_FEATURE_TAG` constant so future developers can see
at a glance which flag governs it.

```python
# Example pattern in main_frame_menu.py
if flags.ai:
    menu.Append(_id_ask_ai, "Ask &Quill\tAlt+Q")
    menu.Append(_id_prompt_library, "Prompt &Library")
    menu.Append(_id_skill_library, "Skill &Library")
    # ... etc
```

### 5.2 Command Palette Filtering

Every command in the `CommandRegistry` gains a `feature_tag: str | None`
field. The palette's search backend calls `is_enabled(cmd.feature_tag)` and
excludes commands whose tag is disabled. Commands with `feature_tag = None`
are always shown (core editing commands).

Feature tag constants are defined in `feature_flags.py`:

```python
FEATURE_AI        = "ai"
FEATURE_REMOTE    = "remote"
FEATURE_QUILLINS  = "quillins"
FEATURE_POWER     = "power_tools"
FEATURE_NOTEBOOK  = "notebook"
```

### 5.3 Keyboard Binding Filtering

The keymap loader skips binding registrations for commands whose feature tag
is disabled. This prevents ghost shortcuts that do nothing from cluttering
`Ctrl+?` discovery output.

### 5.4 CI Gate — `check_feature_tags.py`

A new gate tool (same family as `check_dialog_zorder.py`) walks the command
registry at import time and asserts that every command either has a known
`feature_tag` value or explicitly declares `feature_tag=None`.

Run: `python -m quill.tools.check_feature_tags`
Test: `tests/unit/tools/test_feature_tags.py`

## 6. First-Run Detection and Re-run

### First run

In `MainFrame.__init__`, before any UI is shown:

```python
if not load_settings().setup_wizard_completed:
    from quill.ui.setup_wizard import run_setup_wizard
    run_setup_wizard(parent=None)  # modal, no parent yet
    # settings are now written; reload before building menus
    reload_settings()
```

The wizard window is created as a top-level frame (not a child of MainFrame)
so it can be the application's main window during the setup phase.

### Re-run

`Help > Personalise QUILL...` (command `open_setup_wizard`, always visible
regardless of flags, feature_tag=None). The Help menu is the natural home
for this command: it is where users go when they are figuring out how the
editor works, and "Personalise QUILL" is the direct answer to "how do I
make this editor suit me?" — a discovery question, not a text-operation.
Placing it in Tools would bury a setup action among productivity utilities.

The command opens the wizard in update mode: all pages are pre-filled with
current settings. On Finish, changed flags trigger `_apply_feature_flags()`
to rebuild the affected menu groups and refresh the palette without a restart.

If re-run disables a feature that was previously enabled, a one-time
announcement reads: "AI features have been hidden. You can re-enable them
here at any time."

---

## 7. `_apply_feature_flags()` — Live Rebuild

This method on `MainFrame`:

1. Calls `_rebuild_menus()` (already exists for dynamic menu changes).
2. Calls `command_registry.rebuild_filtered_set(flags)`.
3. Calls `keymap.apply_profile(settings.keymap_profile)`.
4. Calls `announce("Setup complete. [N] features enabled, [M] hidden.")`.
5. Calls `Layout()` on the frame.

All of this is done on the UI thread (it is synchronous and fast).

---

## 8. Accessibility Requirements for the Wizard Itself

1. Every page announces its heading via a live region the moment it becomes
   active (use `wx.adv.Wizard.EVT_WIZARD_PAGE_CHANGED`).
2. Focus on page change lands on the first interactive control (or the Next
   button if the page has no inputs).
3. Show/hide of conditional sub-sections (AI provider details, inline site
   form) must not leave focus on a now-hidden control. After hiding, focus
   moves to the Next button; after showing, focus moves to the first new
   control.
4. All radio groups are wrapped in a `wx.StaticBox` labelled with the question
   text. NVDA reads this as a group name, giving full context.
5. Error states (e.g. blank API key when OpenAI is selected and the user tries
   to advance) use a `wx.adv.NotificationMessage` balloon anchored to the
   offending field. The balloon text is also set as `SetToolTip` on the field
   so it is keyboard-accessible.
6. The summary list on Page 9 uses `wx.LC_REPORT` with explicit column headers
   ("Feature" and "Status") so JAWS and NVDA announce the column name with
   each cell when reading in table mode.
7. Cancel confirmation uses `wx.MessageDialog` (not a raw `ShowModal`) so the
   dialog contract is satisfied and keyboard shortcuts (Enter/Escape) work
   correctly.

## 14. Context-Sensitive Help System

Adapted from ChapterForge's control_help / context_help architecture.
The same design principle applies: the in-app F1 answer and the documented
answer are rendered from identical templates — they cannot say different things.

### 14.1 Keystrokes

| Key       | Command                | Behaviour                                                     |
|-----------|------------------------|---------------------------------------------------------------|
| F1        | Help on This Control   | Shows a small dialog describing the focused control           |
| Ctrl+F1   | Open User Guide        | Opens docs/html/USER_GUIDE.html in the system browser         |
| Shift+F1  | What Can I Do Here?    | Existing document-type context report (kept, enhanced)        |

`F1` is wired as `Help > &Help on This Control\tF1`, matching ChapterForge.
`Ctrl+F1` upgrades the current un-accelerated "Open User Guide" item.
`Shift+F1` upgrades the current un-accelerated "What Can I Do Here?" item.

---

### 14.2 Architecture Overview

```
quill/
  core/
    help/
      topics.json          -- schema: every control, category, template, tokens
      renderer.py          -- token resolution (live + doc), generate_markdown()
  ui/
    context_help.py        -- ContextHelpDialog, ContextHelpMixin
  tools/
    build_docs.py          -- Markdown to HTML doc builder; calls renderer.generate_markdown()
    check_help_coverage.py -- CI gate: stale schema entries flagged, new gaps warned
tests/
  unit/
    tools/
      test_help_coverage.py
docs/
  userguide.md             -- restructured, greatly enhanced (see section 14.7)
  CONTROL_REFERENCE.md     -- auto-generated from topics.json on every doc build
  html/
    USER_GUIDE.html        -- generated
    CONTROL_REFERENCE.html -- generated
```

---

### 14.3 The Schema: `quill/core/help/topics.json`

Identical structure to ChapterForge's `control_help.json`. Categories must
mirror the userguide section headings (see section 14.7) so the auto-generated
CONTROL_REFERENCE.md slots into the guide cleanly.

Topic ID convention: the control's accessible name in lowercase with spaces
replaced by underscores, matching the `SetName()` call on the control.
For dialog controls, prefix with the dialog's accessible name:
`connect_to_ssh_server.host_or_ip_address`.

Tokens work identically to ChapterForge: `kind: setting` resolves against the
user's current settings in-app and against documented defaults in the doc
build. `kind: live` calls a named resolver against the running frame in-app
and uses a `doc_example` string in the doc build.

**QUILL-specific live tokens:**

| Token                | Doc example                         | Resolves to                                   |
|----------------------|-------------------------------------|-----------------------------------------------|
| `word_count`         | 1,204 words                         | current document word count                   |
| `doc_name`           | My Draft.md                         | current document filename                     |
| `browse_mode_state`  | inactive                            | "active" or "inactive"                        |
| `extend_mode_state`  | off                                 | "on" or "off"                                 |
| `ai_provider`        | OpenAI (gpt-4o-mini)                | configured provider + model or "not configured"|
| `copy_tray_filled`   | 3 of 12 slots filled                | count of non-empty Copy Tray slots            |
| `quill_key`          | Caps Lock (setting token)           | the key name from settings.quill_key          |

### 14.4 Name-Keyed Lookup — QUILL's Key Adaptation from ChapterForge

ChapterForge maps ~25 controls via a hand-maintained lambda list. QUILL has
200+ controls across dozens of dialogs. Rather than a growing locator list,
QUILL uses `ctrl.GetName()` as the lookup key directly:

```python
def _find_topic_id(ctrl: wx.Window) -> str | None:
    name = (ctrl.GetName() or "").strip()
    if not name:
        return None
    top = wx.GetTopLevelParent(ctrl)
    if isinstance(top, wx.Dialog):
        prefix = top.GetName().strip().lower().replace(" ", "_")
        return f"{prefix}.{name.lower().replace(' ', '_')}"
    return name.lower().replace(" ", "_")
```

This works because QUILL's dialog contract already requires every interactive
control to carry a meaningful `SetName()`. The help system is a natural
extension of the accessibility requirement already enforced by
`check_dialog_zorder.py` and `dialog_inventory.py`.

### 14.5 ContextHelpMixin

Added to MainFrame's MRO alongside `CopyTrayMixin`, `AbbreviationsMixin`, etc.
Provides focus tracking (solves the menu-focus problem), the F1 handler, and
the Ctrl+F1 user guide opener.

The focus-tracking problem: when the user navigates `Help > Help on This
Control` via the menu bar, focus moves to the menu, and `wx.Window.FindFocus()`
returns the menu bar rather than the control the user wants described.
`_last_focused_ctrl` is updated by `EVT_CHILD_FOCUS` before the menu opens,
giving the handler the correct target. This is identical to ChapterForge's
solution.

```python
class ContextHelpMixin:
    _last_focused_ctrl: wx.Window | None = None

    def _init_context_help(self) -> None:
        self.Bind(wx.EVT_CHILD_FOCUS, self._on_child_focus_for_help)

    def _on_child_focus_for_help(self, evt):
        win = evt.GetWindow()
        if win is not None:
            self._last_focused_ctrl = win
        evt.Skip()

    def on_context_help(self, _evt=None) -> None:
        from quill.ui.context_help import describe_focused, ContextHelpDialog
        focused = wx.Window.FindFocus() or self._last_focused_ctrl
        title, body = describe_focused(self, focused)
        dlg = ContextHelpDialog(self.frame, title, body)
        dlg.ShowModal()
        dlg.Destroy()
        if focused:
            focused.SetFocus()

    def on_user_guide(self, _evt=None) -> None:
        from quill.ui import docs
        docs.open_doc(docs.USER_GUIDE)
        self._announce("Opening the User Guide in your browser.")
```

### 14.8 Two Resolution Paths

**Live (in-app):** `renderer.render_live(topic_id, frame)` resolves tokens
against the running app: document stats, settings, mode flags, feature flags.
Variant selection (e.g. browse mode active vs. inactive) happens at render time.

**Documentation (static):** `renderer.render_doc(topic_id)` resolves tokens
against `doc_example` strings and settings defaults. Both mode variants of a
topic render side-by-side in the docs, clearly labelled. This is what
`generate_markdown()` feeds to `build_docs.py`.

Neither path can drift from the other because they use the same template
strings — only the token values differ.

### 14.9 Fallback Chain

1. Schema lookup by topic ID derived from `ctrl.GetName()` -> `render_live()`.
2. Generic description by `ctrl.GetClassName()` (wxButton, wxCheckBox,
   wxTextCtrl, wxChoice, wxSlider, wxListCtrl, wxSpinCtrl — same table as
   ChapterForge with QUILL-specific wording).
3. Last resort: control name + tooltip + "See the User Guide (Ctrl+F1) for
   a fuller walkthrough."

F1 always returns something useful.

## 15. Translation and Community Localization

**Note:** Neither QUILL nor ChapterForge have any i18n infrastructure today. This
section designs it from scratch, explicitly aligned with the principles used by
NV Access for NVDA — the most widely used screen reader in the world and the
closest peer project to QUILL's community. Where the approach diverges from
NVDA's, the reason is documented.

The goal is a first-class translated experience driven entirely by the community
— no paid localization vendor, no English-only assumptions baked into the
codebase, and a zero-friction path for a translator who knows no Python.

---

### 15.1 NV Access Alignment Principles

NV Access has published and practised the following principles for NVDA
translations. QUILL adopts all of them:

1. **The translation platform must be accessible to blind translators.** A
   translator who uses NVDA or JAWS must be able to contribute entirely via
   keyboard. This principle governs the choice of Crowdin over alternatives
   (see section 15.2).

2. **Speech strings are first-class, not afterthoughts.** Strings that are
   spoken by a screen reader are maintained separately from display strings
   where they differ, and the style guide treats them as a distinct category
   requiring natural spoken language, not just translated labels.

3. **Translation is release-synchronized, not rolling.** Before each release
   a formal "translation call" goes to all language teams with a deadline.
   Languages that miss the deadline or fall below the completeness threshold
   are excluded from that release. This prevents half-translated UI shipping
   silently.

4. **A Translation Coordinator role exists at the project level.** This person
   manages the overall program, onboards language teams, resolves disputes,
   and runs the translation call process. It is a named, accountable role —
   not a vague "community" responsibility.

5. **Translators have standing equal to code contributors.** A translation PR
   is reviewed and merged with the same care as a code PR. Translator names
   appear in the release notes.

6. **Add-on (extension) manifests are translatable.** For NVDA, each add-on's
   manifest has translatable name and description fields. QUILL Quillin
   manifests follow the same pattern (see section 15.6).

7. **Translations live close to the source.** NV Access maintains a dedicated
   `nvda-l10n` companion repository; QUILL uses a `quill/locale/` directory
   in the main repository for simplicity, but the release process follows the
   same branch-per-release discipline.

**Where QUILL diverges from NVDA and why:**
- NVDA's documentation (User Guide, What's New) is translated as whole
  Markdown files by language teams directly in the `nvda-l10n` repo. QUILL
  follows the same approach (section 15.10).
- NVDA uses a custom build pipeline to compile `.mo` files. QUILL uses
  standard `pybabel compile` — same outcome, less custom code.

---

### 15.2 Goals

1. Any QUILL string a user ever reads or hears can be translated.
2. A translator needs only a web browser and knowledge of the target language.
3. A missing translation never crashes QUILL — it falls back to English silently.
4. The CI build catches untranslated strings before they ship.
5. Screen reader announcement text is treated with the same care as visible UI text.
6. The community owns translation quality; maintainers own infrastructure.
7. Translations are version-synchronized; no half-translated build ships.

---

### 15.3 Technology Stack

| Layer               | Technology                          | Reason                                               |
|---------------------|-------------------------------------|------------------------------------------------------|
| String format       | GNU gettext (.pot / .po / .mo)      | Industry standard; every translation tool supports it|
| Extraction          | `pybabel extract` (Babel)           | Understands Python, wxPython label patterns          |
| Compilation         | `pybabel compile`                   | Produces .mo files that Python's gettext loads       |
| Community platform  | **Crowdin** (crowdin.com)           | NV Access standard; proven accessibility for blind translators |
| Runtime loader      | Python `gettext` + `wx.Locale`      | Standard library; no additional runtime dependency   |
| Completeness gate   | `check_translation.py` (new tool)   | Blocks release if a shipped language drops below threshold |

**Crowdin** is the NV Access-aligned choice and the right platform for QUILL:
- NV Access uses Crowdin for NVDA — the most screen-reader-accessible
  translation platform available. A blind translator can contribute using
  only NVDA and a browser, which is a hard requirement for this community.
- Free open-source tier covers public repositories.
- Per-string suggestions, comments, glossary, screenshot context, and history.
- GitHub integration: approved translations open PRs automatically.
- Language teams have a proofreader tier that mirrors the NVDA role structure.
- Translation memory and glossary reduce inconsistency across releases.

**Why not Weblate:** Weblate's keyboard accessibility has improved but Crowdin
has the longer track record with screen reader users specifically. NV Access
made this choice deliberately after testing both; QUILL inherits the benefit
of that research.

### 15.5 String Marking Convention

All user-visible strings in Python source are wrapped with `_()`:

```python
# quill/ui/main_frame_menu.py
file_menu.Append(self._id_new, _("&New\tCtrl+N"))
file_menu.Append(self._id_open, _("&Open...\tCtrl+O"))
```

For plurals (word count, slot count, etc.):

```python
from quill.core.i18n import ngettext
label = ngettext("{n} word", "{n} words", count).format(n=count)
```

For deferred translation (strings evaluated at module load time but
translated at use time):

```python
from quill.core.i18n import lazy_gettext as _l
# Used in settings_specs.py where spec labels are defined at import time
SettingSpec(label=_l("Theme"), ...)
```

**What must NOT be wrapped:**
- Log messages (internal, never shown to users).
- Exception messages caught internally and never displayed.
- AI model IDs, provider names, API endpoints (technical identifiers).
- File format names used as mime types or file extensions.
- Debug output.

**What MUST be wrapped that is often forgotten:**
- `SetName()` accessible name strings — screen readers announce these.
- `SetToolTip()` strings.
- `wx.StaticText` labels.
- Status bar cell text.
- Announcement strings passed to `self._announce()`.
- Error messages shown in message boxes.
- Column headers in `wx.ListCtrl`.
- Button labels.
- Dialog titles.

### 15.6 Quillin Manifest Translation

Following the NVDA add-on manifest pattern, each Quillin's `manifest.json`
has translatable fields. The canonical English values live in the manifest
itself; per-language overrides live alongside it:

```
quillins_bundled/
  ai-writing-prompts/
    manifest.json          -- English: {"name": "AI Writing Prompts", "description": "..."}
    manifest_fr.json       -- French: {"name": "Assistants d'écriture IA", "description": "..."}
    manifest_de.json       -- German
```

The Quillin loader in `main_frame_quillins.py` checks for a
`manifest_{lang}.json` matching the active language and merges only the
`name` and `description` fields from it. All other manifest fields
(permissions, entry_point, version, etc.) always come from the base
`manifest.json`. This mirrors how NVDA add-on translations work: the
structural manifest is untouched; only human-visible strings are overridden.

Crowdin manages these JSON files as a separate "Quillin manifests" component.

### 15.7 The `quill/core/i18n.py` Module

A thin wrapper that isolates the gettext machinery from the rest of the codebase.
All import sites use `from quill.core.i18n import _` rather than importing
`gettext` directly, so the locale-loading strategy can change without touching
200 files.

```python
# quill/core/i18n.py
"""Thin wrapper around Python gettext for QUILL's translation system.

Import only from here — never import gettext directly in UI or core code.
The _ function is installed globally by init_locale() at startup; this
module also exports ngettext and lazy_gettext for plurals and deferred
translation respectively.
"""
from __future__ import annotations

import gettext
import locale
import logging
from pathlib import Path

_LOCALE_DIR = Path(__file__).resolve().parents[1] / "locale"
_DOMAIN = "quill"
_translation: gettext.NullTranslations | None = None
_log = logging.getLogger(__name__)


def init_locale(language: str | None = None) -> str:
    """Load the .mo file for *language* (e.g. 'fr', 'de', 'en').

    Called once from MainFrame.__init__ before any UI strings are evaluated.
    Falls back to the OS locale, then to English if no .mo file is found.
    Returns the language code that was actually loaded.
    """
    global _translation
    lang = language or _detect_os_language()
    try:
        _translation = gettext.translation(
            _DOMAIN, localedir=str(_LOCALE_DIR), languages=[lang]
        )
        _translation.install()
        _log.info("Loaded locale: %s", lang)
        return lang
    except FileNotFoundError:
        # No .mo for this language — silent fallback to English
        gettext.install(_DOMAIN, localedir=str(_LOCALE_DIR))
        _log.info("No locale for %s; using English", lang)
        return "en"


def _(message: str) -> str:
    return _translation.gettext(message) if _translation else message


def ngettext(singular: str, plural: str, n: int) -> str:
    if _translation:
        return _translation.ngettext(singular, plural, n)
    return singular if n == 1 else plural


def lazy_gettext(message: str):
    """Return a lazy proxy that translates when str() is called.
    Use for strings defined at module import time (settings specs, etc.)."""
    class _Lazy:
        def __str__(self): return _(message)
        def __repr__(self): return f"lazy_gettext({message!r})"
    return _Lazy()


def _detect_os_language() -> str:
    try:
        lang = locale.getdefaultlocale()[0] or "en"
        return lang.split("_")[0]  # 'fr_FR' -> 'fr'
    except Exception:
        return "en"
```

### 15.9 Mnemonic Preservation

wxPython uses `&` to mark keyboard mnemonics in menu items and button labels
(`&File` makes Alt+F the shortcut). Translators MUST carry the `&` prefix,
placing it before the appropriate letter for the target language.

The Weblate project is configured with a **mnemonic check** that warns if a
translation drops the `&` marker or places it somewhere that conflicts with
another item in the same menu. The CI `check_translation.py` tool also
validates this.

Example: `&File` (English) -> `&Fichier` (French) -> `&Datei` (German).

The translator style guide documents this requirement with worked examples.

### 15.10 Screen Reader Announcement Translation

This is the hardest part and the most important for QUILL's user base.
`self._announce()` calls in `main_frame.py` generate the spoken text that
NVDA and JAWS read aloud. These strings are first-class translation targets.

All announcement strings are extracted into the `.pot` file the same way as
visible UI strings — they already flow through `self._announce(str)` which
we wrap:

```python
# Before
self._announce(f"Copied to slot {n}")

# After
self._announce(_("Copied to slot {n}").format(n=n))
```

The translator style guide includes a section specifically on announcement
strings: they must be natural spoken sentences, not UI label abbreviations.
A label that reads "Copied: 3" on screen reads awkwardly when spoken; the
announcement should say "Copied to slot 3."

### 15.11 Control Help Schema Translation

The `topics.json` schema (section 14) is English-only. For translated help:

- `quill/core/help/topics.json` is the English canonical source.
- Community translators produce `quill/core/help/topics_fr.json`,
  `topics_de.json`, etc. by translating only the `title` and `template`
  fields. Token names (`{word_count}`, `{ai_provider}`) are never translated
  — they are structural markers.
- The renderer loads `topics_{lang}.json` if it exists, falling back to
  `topics.json`.
- Weblate manages `topics.json` translations as a separate component with
  custom file format (`JSON nested key`).

This keeps the help schema translations outside the gettext system (since
structured JSON is harder to manage in .po format) while still routing
through Weblate for community editing.

### 15.12 Documentation Translation

User guide and control reference translation is Phase 4 work. The approach:

- `docs/userguide.md` is the English canonical source.
- Community language coordinators create and maintain `docs/translations/{lang}/userguide.md`.
- The doc builder (`tools/build_docs.py`) generates HTML for each language
  present in `docs/translations/`.
- The in-app "Open User Guide" command (`on_user_guide()`) opens the file
  matching `self._active_language` if it exists, else falls back to English.

Documentation translations are NOT managed through Weblate (Weblate is for
structured strings; Markdown prose is better handled as whole-file GitHub
contributions). Language coordinators own their language's docs directly.

```bash
# 3. Update all existing .po files with the new string
pybabel update -i quill/locale/quill.pot -d quill/locale -D quill

# 4. Weblate picks up the new string automatically via GitHub sync
# 5. Translators see the new string as "Needs translation" in Weblate
```

**Compiling translations for a test build:**

```bash
pybabel compile -d quill/locale -D quill
```

**Adding a new language (language coordinator starts a new locale):**

```bash
pybabel init -i quill/locale/quill.pot -d quill/locale -D quill -l fr
# Creates quill/locale/fr/LC_MESSAGES/quill.po
# Commit and push; Weblate picks up the new language component
```

### 15.15 CI Gate — `check_translation.py`

```bash
python -m quill.tools.check_translation [--lang fr] [--threshold 80]
```

**Checks performed:**

1. **POT is current:** Run `pybabel extract` in dry-run mode and diff against
   committed `quill.pot`. Fails if strings have been added to source but the
   `.pot` was not regenerated. Prevents the silent drift where developers
   forget to run `pybabel extract`.

2. **Completeness threshold:** For each language in `quill/locale/`, compute
   `translated / total` as a percentage. Fail if any shipped language is below
   the configured threshold (default 80%). Languages below threshold are
   demoted to "preview" status in the UI language selector (shown with a
   warning: "This translation is incomplete — some text will appear in
   English.").

3. **Mnemonic check:** Every translated string that contains a `&` mnemonic
   in the source must contain exactly one `&` in the translation.

4. **Placeholder check:** Every translated string that contains `{variable}`
   placeholders must contain the same set of placeholder names in the
   translation. A translator who deletes `{n}` from a plural string would
   cause a runtime `KeyError`.

5. **Empty string check:** No translated string may be empty. Empty strings
   in `.po` files mean "fall back to source language" in gettext — but QUILL
   uses `NullTranslations` fallback, so an empty translation would display
   as blank.

```
python -m quill.tools.check_translation
OK - quill.pot is current (247 strings).
OK - fr: 231/247 translated (93.5%) -- above 80% threshold.
OK - de: 198/247 translated (80.2%) -- above 80% threshold.
Warning - es: 61/247 translated (24.7%) -- below threshold, will ship as preview.
OK - All mnemonic checks passed.
OK - All placeholder checks passed.
```

### 15.16 Crowdin Project Setup and Community Roles

The `docs/translating.md` guide walks a new language team through joining.
The setup follows the NV Access model directly.

**Crowdin project configuration:**

1. Project created at crowdin.com/project/quill (free open-source tier).
2. Source branch: `main`. Integration: auto-PR on approved translation.
3. Three Crowdin components:
   - `UI strings` — source `quill/locale/quill.pot`,
     target `quill/locale/{lang}/LC_MESSAGES/quill.po`
   - `Context-sensitive help` — source `quill/core/help/topics.json`,
     target `quill/core/help/topics_{lang}.json` (JSON format)
   - `Quillin manifests` — source each `manifest.json`,
     target `manifest_{lang}.json`
4. Glossary pre-populated with QUILL product terms (Copy Tray, Ask Quill,
   Quillin, Quick Nav, Skill Library, Prompt Library).
5. Screenshots uploaded for every dialog and menu item so translators have
   context without running the app.

**Role hierarchy (NV Access model, four tiers):**

| Role                     | Responsibilities                                             | Account needed |
|--------------------------|--------------------------------------------------------------|----------------|
| **Translator**           | Suggests translations in Crowdin                            | Crowdin only   |
| **Proofreader**          | Approves or rejects suggestions for their language          | Crowdin only   |
| **Language coordinator** | Manages proofreaders, owns language quality, reviews PRs    | GitHub + Crowdin |
| **Translation coordinator** | Project-level role; runs translation calls, onboards teams, resolves disputes | GitHub + Crowdin |

The Translation Coordinator is a named maintainer role in `MAINTAINERS.md`,
listed alongside the code maintainers. It is not a "community volunteer"
position — it requires a person who commits to being available before each
release.

**Translation calls (NV Access model):**

Before each QUILL release, the Translation Coordinator sends a translation
call to all language coordinators with:
- The deadline (typically two weeks before the release branch cut).
- The list of new or changed strings since the last release.
- The current completeness percentage for each language.
- The threshold required to ship (default 90% for established languages,
  70% for new languages in their first release).

Languages that miss the deadline or fall below threshold are excluded from
that release. The startup wizard's language selector does not show excluded
languages. This is documented in `docs/translating.md` so language teams
know the stakes.

The repository's `CONTRIBUTING.md` gets a "Translating QUILL" section that
links to Crowdin, describes all four roles, and explains the translation call
process.

### 15.17 Translator Style Guide

`docs/TRANSLATION_STYLE_GUIDE.md` covers:

**Tone and register:**
- QUILL speaks to the user in the second person, directly. "Your document",
  "Press this key", not "The user's document" or "One presses this key."
- Match the register of the target language's screen reader ecosystem.
  In French, use "vous" (formal) unless the target language community
  explicitly votes for "tu" (informal).

**Mnemonics (`&` markers):**
- Every menu item and button label has exactly one `&`.
- Place the `&` before a letter that is unique within the menu or dialog.
- Include a worked example for each language in the guide.

**Keyboard shortcuts:**
- Modifier key names (Ctrl, Alt, Shift) are NOT translated — they appear as
  printed on the keyboard. `Ctrl+F` stays `Ctrl+F` in all languages.
- Mnemonic letters in `\t` accelerator strings (e.g. `&Open...\tCtrl+O`)
  must match the `&` letter: `&Ouvrir...\tCtrl+O` in French.

**Announcement strings (spoken text — NV Access principle):**
- Speech strings are a distinct category from display strings. They are marked
  in the `.pot` file with a `#. SPEECH:` extracted comment so Crowdin can
  filter them as a group for translation review.
- Translate for natural speech, not literal label equivalence. "3 words
  selected" on screen becomes "3 words selected" in speech — but "Copied to
  slot 3" on screen might need to be "Copied to Copy Tray slot 3" in speech
  because the context is lost when read aloud without the visual UI.
- Do not abbreviate in speech strings. "AI" should be spelled out as the
  language's equivalent ("intelligence artificielle" in French) unless the
  abbreviation is universally understood in that language's tech community.
- Test every announcement string with NVDA set to the target language. NV
  Access recommends testing with a native speaker using their preferred screen
  reader configuration — not just by reading the .po file.

**Plural forms:**
- Follow the language's actual plural rules. French has two forms
  (0 and 1 are singular). Russian has four. The `.po` file `Plural-Forms`
  header must match.
- Do not invent plural forms. Reference
  https://www.gnu.org/software/gettext/manual/html_node/Plural-forms.html

**Technical terms:**
- A glossary in Weblate defines terms like "Copy Tray", "Quick Nav",
  "Quillin", "Skill Library", "Prompt Library". These are QUILL product
  names and may be left in English or translated — the language coordinator
  decides and the glossary entry records the decision.

### 15.18 Language Priority for Initial Sprint

Languages are selected based on the screen reader user community size,
existing open source translator availability, and right-to-left complexity
(RTL languages require additional layout work and are Phase 5).

**Tier 1 — pilot (target: ship with QUILL 1.0):**
- French (fr)
- German (de)
- Spanish (es)

**Tier 2 — close follow-on:**
- Portuguese / Brazilian Portuguese (pt_BR)
- Japanese (ja)
- Italian (it)

**Tier 3 — RTL, requires layout work:**
- Arabic (ar)
- Hebrew (he)

For RTL languages, the startup wizard's language selector displays a note:
"Right-to-left layout support is in progress. Some dialogs may not render
correctly yet." This is honest and prevents RTL users from encountering
broken layouts silently.

### 15.22 Implementation Phases

**Phase 1 — Infrastructure (no visible change to users)**
1. Add `quill/core/i18n.py` with `_()`, `ngettext()`, `lazy_gettext()`,
   `init_locale()`.
2. Add `babel.cfg`.
3. Add Babel to dev extras in `pyproject.toml`.
4. Run `pybabel extract` to generate `quill/locale/quill.pot` (English source).
5. Call `init_locale()` in `MainFrame.__init__`; add `language` setting.
6. Add `check_translation.py` gate and test.
7. Verify: app launches, English strings unaffected, gate passes.

**Phase 2 — String Marking Sprint**
Sweep all user-visible strings in `quill/ui/` and `quill/core/`:
- `main_frame.py` — menus, status bar, message boxes, announcements.
- All dialog files — `SetName()`, `label=`, titles, button text.
- `settings_specs.py` — spec labels via `lazy_gettext()`.
- `stability/` — user-facing error messages.
- `prism_bridge.py` / announcement strings.

After each file, run `pybabel extract` and verify `.pot` grows correctly.
Estimated scope: 600-900 translatable strings.

**Phase 3 — Pilot Languages (NV Access-aligned launch)**
1. Appoint Translation Coordinator; add to `MAINTAINERS.md`.
2. Set up Crowdin project with three components (UI strings, help topics,
   Quillin manifests) for French, German, Spanish.
3. `pybabel init` for fr, de, es; push `.po` stubs to repo.
4. Post translation call to QUILL forums, ACB/NFB/RNIB/accessibility mailing
   lists, and national blind/VI organizations in target locales.
5. Translators work in Crowdin; proofreaders approve; auto-PRs land on main.
6. Run `pybabel compile`; test QUILL with `language = "fr"` in settings.
7. Language coordinators do a screen-reader walkthrough (NVDA with French
   language pack, verifying announcement strings are natural spoken French).
8. Run translation call process: set deadline, communicate threshold, exclude
   any language that misses the bar from the first public release.

**Phase 4 — Control Help Schema Translation**
1. Document the `topics_{lang}.json` format.
2. Weblate JSON component for `topics.json`.
3. French pilot: translate 50 highest-traffic topics first (main editor,
   status bar, copy tray, ask quill dialog).

**Phase 5 — Documentation Translation**
1. Create `docs/translations/` structure.
2. French pilot: translate `docs/userguide.md` to `docs/translations/fr/userguide.md`.
3. Update `build_docs.py` to generate HTML per locale.
4. Wire `on_user_guide()` to open locale-matched HTML.

**Phase 6 — RTL Language Support**
1. Audit all dialogs for fixed-direction layout assumptions (left-aligned
   labels, icons on left, etc.).
2. Add `wx.Layout_RightToLeft` support to dialog builders.
3. Invite Arabic and Hebrew language coordinators.
